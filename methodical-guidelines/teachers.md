# Методические указания преподавателю

## 1. Назначение

Документ помогает подготовить, провести и оценить девять лабораторных работ без дополнительных устных пояснений авторов комплекта. Постановки заданий находятся в [карточках лабораторных работ](../assessments/laboratory-works/README.md), общая шкала — в [модели оценивания](../assessments/grading-policy.md), связь с результатами — в [матрице измеримости](../docs/measurement-matrix.md).

## 2. Подготовка к лабораторным работам

До начала соответствующего модуля преподаватель:

1. проверяет запуск рекомендуемого окружения и доступность моделей;
2. готовит входные изображения, размеченные примеры и видео, если они требуются;
3. сообщает поддерживаемые версии Python и библиотек;
4. предоставляет доступную альтернативу платному сервису, GPU или закрытым данным;
5. публикует сроки, канал сдачи и критерии до выдачи задания;
6. не изменяет обязательные критерии после начала работы без одинакового письменного уведомления всей группы.

Студент сдаёт исполняемый ноутбук или код, отчёт и предусмотренные заданием артефакты. Для оценивания фиксируется версия файла или commit hash.

## 3. Общий порядок оценивания

- Проверить воспроизводимость основного сценария на предоставленных или согласованных данных.
- Выставить баллы отдельно по критериям конкретной работы. Если детальная рубрика отсутствует, использовать единую рубрику `50/25/15/10` из модели оценивания.
- Провести короткую индивидуальную защиту продолжительностью 5–7 минут: попросить объяснить ключевое решение, изменить один параметр и интерпретировать результат.
- Не считать визуально привлекательный результат заменой корректному эксперименту и метрикам.
- Не снижать балл за отсутствие личного оборудования, если студент использовал утверждённую альтернативную среду.
- Зафиксировать замечания и баллы по критериям так, чтобы решение можно было проверить при апелляции.

Работа направляется на доработку, если не запускается её обязательный сценарий, отсутствует обязательный артефакт, студент не может объяснить собственный код или не подтверждён некомпенсируемый индикатор.

| `LAB-03` | согласованный датасет или требования к пользовательскому набору | корректное разбиение, адаптация MobileNetV2, метрики и анализ ошибочных классов | оценивание на обучающей выборке, неверная нормализация предобученной модели, отсутствие анализа ошибок |

| `LAB-05` | примеры предсказаний и эталонных меток для классификации и детекции | корректность confusion matrix, precision/recall/F1, ROC-AUC, IoU/mAP и интерпретации | ROC-AUC по жёстким меткам вместо scores, смешение macro/micro averaging, mAP без определения IoU |
| `LAB-06` | не менее 10 размеченных изображений либо задание на их ручную разметку | раздельное влияние confidence и IoU NMS, precision/recall/mAP, время инференса | изменение только confidence, подсчёт боксов вместо качества, сравнение скорости без одинаковой конфигурации |
| `LAB-07` | рабочая версия TensorFlow/Keras и ограниченный по времени режим обучения | полный цикл GAN, кривые потерь, разнообразие выборки, абляция стабилизации, сохранение генератора | только запуск готового инференса, отсутствие фиксированных условий сравнения, оценка GAN через Grad-CAM |
| `LAB-08` | GPU-среда или заранее проверенная облегчённая альтернатива; доступ к моделям | одинаковый seed, влияние steps/CFG/scheduler/strength, inpainting, ControlNet и ограничения метода | сравнение при разном шуме, импорт scheduler без его замены, несогласованные размеры маски, требование личного GPU |
| `LAB-09` | короткое видео с эталонными ID либо протокол разметки 100 кадров; локальные PySpark и Airflow | ID switches и IDF1/MOTA, FPS, схема результатов, PySpark-агрегация, зависимости и идемпотентность DAG | субъективная оценка трекинга без эталона, pandas вместо PySpark, DAG без запуска, повторное накопление дублей |

Для `LAB-08` дополнительно применяется [чек-лист экспресс-проверки](teachers/lab-08-code-review-checklist.md). Эта работа имеет форсайт-направленность: оценивается осмысленное применение и критический анализ современных генеративных пайплайнов, а не разработка новой диффузионной архитектуры.

## 5. Калибровка оценивания

Перед первым проведением курса преподавателям рекомендуется совместно оценить по одной модельной работе низкого, среднего и высокого качества. Следует согласовать:

- какие ошибки являются критическими;
- что считается достаточным аналитическим выводом;
- как оценивать частично работающий код;
- какие альтернативные среды и данные эквивалентны;
- какие ответы подтверждают самостоятельность.

Если работу проверяют несколько преподавателей, расхождение более чем на 10 баллов обсуждается по отдельным критериям, а не посредством усреднения без объяснения.

## 6. Обратная связь, пересдача и особые случаи

- Обратная связь содержит баллы по критериям и перечень конкретных исправлений.
- При пересдаче студент демонстрирует исправленные обязательные задания и повторно проходит защиту по затронутым результатам.
- При техническом сбое по вине инфраструктуры предоставляется повторный запуск или эквивалентная среда без штрафа.
- Использование внешнего кода и генеративного ИИ допускается только с раскрытием источника и объяснением внесённых изменений. Сокрытие заимствования рассматривается по локальным правилам академической добросовестности.
- При специальных образовательных потребностях формат демонстрации может быть адаптирован без снижения проверяемого уровня результата.

Апелляция проводится по локальному регламенту университета на основе опубликованной рубрики, сохранённой версии работы и комментариев преподавателя.
